# LOOP Creator OS · 中文用户手册

[中文](README.zh-CN.md) | [English](README.md)

LOOP Creator OS 是面向红人营销团队的首次建联 MVP。它把达人画像、项目分组、英文邮件模板、投递通道、人工回信邮箱、可选 AI 个性化、人工审批和安全发送放在同一个中文工作台中。当前版本只负责批量首封建联，达人回复后由团队在原邮箱中人工沟通。

当前版本适合在 Windows 本机由运营人员使用。示例项目是西班牙 TikTok Shop 固定 €20 合作邀约；仓库不包含真实邮箱密码或模型 API Key，克隆后不会自动发送邮件。

## 五分钟开始使用

### 1. 准备运行环境

请保留完整仓库目录，不要只复制 EXE。电脑需要：

- Windows 10 或 Windows 11；
- Node.js `22.13.0` 或更高版本；
- 首次安装依赖时可访问 npm；
- 一个可用于测试的邮箱账户。

### 2. 双击启动

进入以下目录并双击 `LOOP-Creator-OS-MVP.exe`：

```text
scaffolds/kol-email-outreach/ui/release/
```

启动器会自动：

1. 查找兼容的 Node.js；
2. 安装缺失的依赖；
3. 启动中文工作台和本地发送网关；
4. 打开 <http://127.0.0.1:8877/>；
5. 在右下角托盘常驻，方便重新打开或安全退出。

首次启动可能需要几分钟。当前 EXE 未做商业代码签名；如果 Windows SmartScreen 提示风险，请先确认文件来自本仓库，再选择“更多信息”继续运行。

### 3. 完成初始化向导

第一次进入工作台会看到初始化提示：

1. 选择“纯模板”或“AI 个性化开场”；
2. 选择“个人邮箱池”或“企业客户邮箱”，添加并验证至少一个投递通道和人工 Reply-To 邮箱；
3. 只有选择 AI 模式时才需要填写模型 API Key。

初始化不会发送邮件。所有真实发送仍然需要逐封审批和第二次 LIVE 确认。

### 4. 先用内部邮箱测试

不要直接用真实达人验证首发流程。建议先导入 1–3 个团队内部邮箱，完成一次生成、审批和发送，并核对实际收件情况与 Message-ID。

## 两种邮件生成方式

| 模式 | 是否需要模型 API | 每位达人内容 | 适合场景 |
| --- | --- | --- | --- |
| 纯模板（不调用 AI） | 否 | 使用相同商业模板，只替换发件人签名 | 固定 Offer、快速测试、严格统一文案 |
| AI 个性化开场 | 是 | 根据达人公开画像生成不同英文开场 | 需要提高相关性和回复率的项目 |

纯模板模式不会偷偷调用模型。如果模板中保留独立的 `{{personalized_hook}}` 段落，生成审批邮件时会安全移除该段。

AI 模式只允许模型生成一段个性化开场。报价、样片链接、交付要求、CTA 和签名始终由项目模板控制，模型不能改写商业条款。

## 配置投递通道和邮件池

打开左侧“投递与邮箱”，可以添加多个发件身份，再按项目或单个达人分配。

### 个人邮箱池

适用于已经准备好的 Gmail、Outlook 或其他邮箱账号。每个邮箱必须独立完成 OAuth 官方授权或 SMTP 应用密码连接，不能只靠邮箱地址发信。系统服务端强制每个个人邮箱每日最多 100 封、同一邮箱两次尝试至少间隔 30 秒；建议新邮箱从每天 10–25 封开始，不要把上限当成推荐值。

### 企业客户邮箱

面向公司客户时，让客户授权自己的 Google Workspace、Microsoft 365、品牌 SMTP 或已验证域名 SES。邮件应由真实品牌身份发送；不得用平台公共邮箱冒充客户。普通企业邮箱在 LOOP 内每日最多 500 封、同一邮箱至少间隔 10 秒。

### Amazon SES API（企业域名链路）

1. 在 AWS SES 中验证真实发件域名，完成 DKIM，并申请生产发送权限及所需额度；
2. 创建只有 `ses:GetAccount`、`ses:GetEmailIdentity` 和 `ses:SendEmail` 权限的 AWS 凭据；
3. 在 LOOP 中填写 AWS Region、Access Key ID、Secret Access Key、发件邮箱和人工 Reply-To；
4. 保存后点击“测试连接”。该操作只验证权限和发件身份，不会发送测试邮件；
5. AWS 凭据只保存在本机发送网关内存中，重启后需要重新输入。

SES API 与 SES SMTP 是同一项服务的两种接入方式。LOOP 只允许 SES 进入企业/域名链路，测试连接会要求域名身份和 DKIM 都已就绪；SES 在 LOOP 内每日最多 5,000 封、同一通道至少间隔 2 秒。额度仍受 AWS 实际批准值约束。

### Gmail / Outlook

填写显示名称、邮箱地址和每日上限，然后完成服务商官方授权。要使用“只填邮箱后授权”，本机网关必须先配置对应的 OAuth 应用；如果页面显示服务商尚未配置，可先使用自定义 SMTP，或让管理员按[工作台技术说明](scaffolds/kol-email-outreach/ui/README.md#本地运行)配置 OAuth。

### 自定义 SMTP

填写邮件服务商提供的 SMTP Host、端口、TLS 方式和应用密码。不要使用日常登录密码；具体参数以邮箱服务商后台说明为准。

每个发件身份都必须通过“测试连接”才能用于正式发送。AWS 凭据、OAuth Token、SMTP 密码和 AI API Key 都是只写字段，只保存在当前本地网关内存中，重启后需要重新授权或重新输入。

## 导入达人数据

在“项目中心”点击“导入达人”，支持：

- CSV；
- JSON；
- JSONL；
- 画像 MCP 返回的常见 JSON 包装结构。

系统会统一识别达人 ID、账号、主页、联系邮箱、国家/城市、粉丝数、均播、互动率、分类标签、简介和近期内容证据，并执行账号与邮箱去重。

单次最多导入 1,000 位达人、文件不超过 5 MB。缺少有效账号或联系邮箱的记录不会进入可发送达人库，界面会显示拒绝原因。

## 创建项目

每个 Campaign 建议建立独立项目：

1. 填写项目名称、品牌、市场、报价和币种；
2. 选择本项目要联系的达人和总数量；
3. 选择默认发件邮箱；
4. 选择纯模板或 AI 个性化；
5. 编辑英文主题和正文；
6. 保存项目。

项目支持逐达人覆盖发件邮箱。修改模板、达人名单、生成模式或发件分配，会自动使尚未发送的旧审批失效，避免用旧内容误发。

## 生成、审批与发送

完整流程如下：

1. 进入“审批与发送”；
2. 生成模板审批批次或 AI 个性化审批批次；
3. 逐封核对 From、To、Subject、Body 和人工警告；
4. 输入完整项目名，批准并锁定批次；
5. 再次输入项目名并勾选 LIVE 确认；
6. 邮件进入服务端顺序队列；
7. 在“发送结果”查看每封邮件状态；达人回复进入 Reply-To 邮箱，由人工处理。

浏览器不会循环调用邮件服务商。它只提交一次锁定批次，服务端逐封发送，因此重复点击或短暂网络波动不会创建第二组邮件。

## 发送安全机制

- 审批后锁定 From、To、Subject、Body 和 SHA-256 快照；
- 同一项目、达人和步骤使用稳定幂等键；
- 所有项目共享发送锁，避免多个项目并发消耗同一邮箱额度；
- 每封邮件发送前记录 `sending` 和唯一尝试ID，明确成功后保存服务商 Message-ID；
- SMTP 超时或结果无法确认时标记 `delivery_unknown`；
- 出现未知投递后全局熔断，不会自动重试；
- 个人、企业和 SES 使用不同的服务端日上限与最小发送间隔，前端不能绕过；
- 所有模板必须保留英文停止联系语；
- 拒绝、退订、投诉或退信可在“发送结果”一键加入全局停止联系名单，所有项目发送前再次拦截；
- MVP 默认不主动注入开信像素或改写点击链接；
- 自动 Follow-up 在当前西班牙固定报价项目中关闭。

遇到 `delivery_unknown` 时不要再次点击发送。先检查邮箱 Sent 文件夹或邮件服务商记录，再人工确认是否已投递。

## 常见问题

### 双击 EXE 后没有窗口

工作台运行在浏览器中，不是传统桌面窗口。先检查右下角托盘是否有 LOOP 图标，再打开 <http://127.0.0.1:8877/>。

### EXE 显示无法启动

按以下顺序检查：

1. EXE 是否仍在 `ui/release` 目录，且完整仓库文件没有被移动；
2. Node.js 是否达到 `22.13.0`；
3. 查看 `scaffolds/kol-email-outreach/ui/server-data/launcher.log`；
4. 从托盘选择“退出并停止服务”，再重新双击 EXE。

### 页面显示“发送网关离线”

说明中文界面仍在，但本地发送服务未运行。通过托盘完整退出后重启 EXE；不要把 `127.0.0.1:8878` 暴露到公网。

### Gmail 或 Outlook 无法授权

当前 OAuth 需要管理员预先配置服务商应用。如果初始化状态显示未配置，可以改用自定义 SMTP，或参考[OAuth 配置说明](scaffolds/kol-email-outreach/ui/README.md#本地运行)。

### 重启后密码或 API Key 消失

这是当前 MVP 的安全设计：敏感凭据只保存在进程内存中，不写入项目数据库、日志或仓库。重新输入即可。

### 无法生成审批批次

请确认：

- 项目已选择达人；
- 达人数等于项目目标数量；
- 项目已选择默认发件邮箱；
- 对应邮箱已通过连接验证；
- AI 项目已配置 API Key；
- 达人联系邮箱有效且不重复。

### 如何停止服务

右键 Windows 右下角 LOOP 托盘图标，选择“退出并停止服务”。不要只关闭浏览器标签页，因为后台服务仍可能运行。

## MVP 当前边界

当前已经实现：

- 达人画像文件导入；
- 项目与达人分配；
- 多发件邮箱池；
- 纯模板和 AI 个性化两种生成方式；
- 逐封人工审批；
- 安全发送队列、幂等和未知投递熔断；
- 发送结果和本地审计记录；
- Windows EXE 启动器。

当前尚未完整实现：

- 自动读取 Gmail/Outlook 收件箱并同步回复（当前明确交由人工处理）；
- 报价自动回写和完整后续邮件工作流；
- 多人共享的云端发送网关和正式 Secret Manager；
- EXE 商业代码签名与完全独立的免 Node 安装包。

界面明确显示“回复转人工”，不会展示伪造回复，也不会自动发送 Follow-up。

- [ ] 使用团队内部邮箱完成 1–3 封测试；
- [ ] 发件邮箱、显示名称和 Reply-To 正确；
- [ ] 英文主题、报价、样片链接和签名正确；
- [ ] AI 项目逐封检查个性化开场和证据警告；
- [ ] 重复点击不会产生第二封；
- [ ] “发送结果”能看到状态和 Message-ID，测试回复能进入正确的人工邮箱；
- [ ] 明确团队处理 `delivery_unknown` 的人工流程；
- [ ] 正式分发前完成域名发信认证、额度策略和 EXE 代码签名。

## 开发者运行与验证

不使用 EXE 时，可以在 PowerShell 中启动：

```powershell
cd scaffolds/kol-email-outreach
npm ci

cd ui
npm ci
npm run mvp
```

重新编译 Windows 启动器：

```powershell
cd scaffolds/kol-email-outreach/ui
npm run package:windows
```

运行测试：

```powershell
cd scaffolds/kol-email-outreach
node --test
```

测试不会连接真实 SMTP，也不会发送邮件。
